# Arquitectura del Sistema y Motor CSP

## 1. Arquitectura Base (Backend en Cloud Functions)

El proyecto utiliza una **Arquitectura Hexagonal (Puertos y Adaptadores)** dentro de Firebase Cloud Functions. Esto aísla la lógica matemática pura (el Motor CSP) de las dependencias externas como Firestore o el protocolo HTTP.

```text
functions/src/
├── domain/            # Capa interna pura
│   ├── model/         # Interfaces TS (Course, Teacher, Schedule)
│   ├── errors/        # Domain exceptions
│   └── ports/         # Interfaces de repositorios (ICourseRepo)
├── application/       # Capa de Orquestación
│   └── usecases/      # Motor CSP y Casos de Uso (ej. generateSchedule)
└── infrastructure/    # Adaptadores Tecnológicos
    ├── firestore/     # Implementación concreta de Repositorios
    └── http/          # Handlers de Functions (onRequest, onCall)
```

## 2. El Motor CSP (Constraint Satisfaction Problem)

El núcleo del sistema es un algoritmo inteligente escrito puramente en TypeScript diseñado para resolver el problema de asignación de horarios utilizando técnicas avanzadas de Inteligencia Artificial (CSP).

### 2.1 Algoritmo de Backtracking (Búsqueda por Retroceso)

El motor intenta asignar un valor a una variable. Si llega a un punto donde una variable no tiene valores legales disponibles, el motor **retrocede** (backtrack) a la variable anterior y prueba una asignación diferente. Es un proceso recursivo de árbol de decisiones.

### 2.2 MRV Heuristic (Minimum Remaining Values)

Para optimizar dramáticamente el árbol de búsqueda (y evitar un tiempo de ejecución exponencial), el algoritmo no elige variables al azar. En cada paso, el motor selecciona la **variable (curso/sección) que tiene la menor cantidad de horarios/aulas legales disponibles**.

> _Al tratar las decisiones más restringidas primero, el algoritmo detecta fracasos tempranamente, podando ramas inútiles del árbol._

### 2.3 Forward Checking

Cada vez que el motor asigna un horario y aula a un curso, propaga esta restricción hacia adelante. Examina todas las variables no asignadas y **elimina temporalmente** de sus dominios el horario/aula que acaba de ser ocupado.

- Si el dominio de alguna variable futura se queda vacío (cero opciones legales), el motor sabe inmediatamente que la ruta actual es un callejón sin salida y realiza un backtracking sin perder tiempo explorando más a fondo.

## 3. Arquitectura Frontend (React + Zustand)

El cliente utiliza React 19 empoderado con **Zustand** para un manejo de estado global atómico y libre de renderizados innecesarios.

- **Diseño Glassmorphism:** Implementado nativamente usando variables CSS en `index.css` y utilidades de Tailwind.
- **GSAP:** GreenSock Animation Platform se utiliza en los componentes principales (como la Landing Page) para proveer micro-interacciones avanzadas, transiciones complejas (staggering) y manipulación del DOM altamente eficiente.
